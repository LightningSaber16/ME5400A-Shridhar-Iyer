01

Project overview and research question

This project studies a mobile surveillance robot that detects and pursues an intruder in a cluttered environment using a Braitenberg vehicle control architecture — sensors wired directly to motors with no planning, no map, and no global state. The core appeal is low-latency reaction, minimal computational overhead, and deployability without infrastructure.

Core research question

How does obstacle density affect the performance of a reactive Braitenberg pursuit controller in cluttered security environments, and to what extent can dynamically adapted channel weights recover lost performance?

The project title is Quantitative Evaluation of Braitenberg-Based Pursuit Behaviour in Cluttered Surveillance Environments. The work has three specific objectives: characterise the relationship between obstacle density and pursuit performance for the baseline controller; identify the failure modes that cause degradation at high density; and evaluate whether a reinforcement-learning policy that adapts the channel weights in real time can recover performance without sacrificing the reactive character of the system.

Academic framing The Braitenberg architecture is not presented as inferior to learning methods — it is the foundation. RL is introduced as a targeted augmentation that learns one specific thing (how aggressively to pursue vs. avoid, given the current sensor state), while all other behaviour remains reactive and interpretable.
02

Braitenberg vehicle theory

Valentino Braitenberg's 1986 thought experiments described vehicles whose behaviour — fear, aggression, curiosity — emerges entirely from the topology and sign of sensor-to-motor connections. No representation, no reasoning. Apparent purposiveness arises from coupling.

The two channels used in this project

The robot superimposes two Braitenberg channels additively every step.

Pursuit — Vehicle 2b (excitatory, crossed). Each motor is driven by the opposite target sensor. A target to the left activates the left sensor, which drives the right motor, steering the robot toward the target. The motor command is:

v_Lpursuit = α · g_t · s_R     v_Rpursuit = α · g_t · s_L

Avoidance — Vehicle 3a (inhibitory, crossed). Proximity sensors aggregate into left and right loads. A strong reading on the left inhibits the right motor, turning the robot away. The baseline robot uses a simple max per hemisphere:

P_L = max(p_i : left half)    P_R = max(p_i : right half)
v_L = v_Lpursuit − β · g_p · P_R     v_R = v_Rpursuit − β · g_p · P_L

Motor speeds are clamped to [0, v_max]. The nominal values are α = 1.0, β = 1.2 (avoidance weighted slightly higher than pursuit — collisions are categorically worse than a slower capture).

Why the weights matter

The ratio α:β is the primary lever the RL policy learns to adjust. At low density, a high α relative to β produces fast capture. At high density, the same ratio causes the robot to ram into obstacles. The optimal ratio is a function of the current local sensor state — which is exactly what the policy learns to compute.

03

Simulation environment and physical grounding

The simulation is a 2D top-down environment implemented in Python with Pygame for visualisation. This is a standard and defensible abstraction for differential-drive pursuit in robotics literature — all relevant dynamics (sensing, avoidance, pursuit) occur in the horizontal plane.

Scale and physical references

All parameters are grounded against real hardware rather than invented arbitrarily. The sensor configuration (8 IR proximity sensors at 45° intervals) matches the e-puck robot, which was designed specifically for Braitenberg vehicle research. The arena (800 × 800 px at ≈ 1 px/cm) represents an 8 × 8 m space — a building lobby, open-plan office bay, or warehouse section.

They represent columns, pillars, and standalone furniture. Non-circular geometries (walls, shelving) are a stated future extension in the report scope section.
04

The three-stage experimental pipeline

The project compares three controllers of increasing sophistication on identical obstacle layouts and intruder trajectories. Each stage adds one contribution on top of the previous.

1Pure Braitenberg

Fixed α = 1.0, β = 1.2. Max-aggregation avoidance. No forward bias. No stuck recovery. Pure sensor-to-motor coupling.

This is the baseline — demonstrates where reactive control fails.

robot.py run_experiments.py → results/
2RL Adaptive Weights

Same robot, but α and β adjusted each step by a trained policy network. Policy outputs (Δα, Δβ) based on current sensor state.

RL contribution — can the policy recover performance at high density?

hybrid_robot.py train_rl.py → models/stage2/ run_hybrid_experiments.py → results_hybrid/
3Improved + RL

Improved base robot (cosine blending, forward bias, motor floor, stuck recovery) with the same RL weight adaptation on top.

Combined system — engineered improvements + learned adaptation.

robot_improved.py hybrid_robot_improved.py train_rl.py --improved → models/stage3/ run_improved_experiments.py → results_improved/

The key design principle: all three controllers are evaluated on identical obstacle layouts and intruder trajectories, generated from the same seed hierarchy. Any difference in results is attributable to the controller, not the environment.

What changes between stages

Stage 1 → 2: only the α and β values change — everything else (sensing, kinematics, collision) is identical. Stage 2 → 3: the base robot gains four engineered improvements (cosine-weighted proximity blending, forward bias, motor floor, stuck recovery). The RL algorithm and policy architecture are the same for both Stages 2 and 3, so any performance gap between them is directly attributable to the improved base controller.

All tunable constants. Single source of truth — every other file imports this.

Target agent. Three modes: bounce (constant velocity), random_walk (periodic heading change), waypoints (follows a sequence).

Visualisation (Pygame)

All Pygame drawing. RGBA overlay for transparent sensor arcs. Proximity rays colour by intensity. HUD readout.

main.py

Live interactive simulation. Keys: S (sensors), R (reset), Space (pause), Q (quit). Wires logger to live session.

RL components

rl_policy.py

PolicyNetwork: 12 → 32 (ReLU) → actor head (2 outputs) + critic head (1 output). Pure numpy. Save/load via .npz.

hybrid_robot.py

Stage 2 controller. Subclasses BraitenbergRobot, queries policy for (Δα, Δβ), patches config temporarily, restores after.

hybrid_robot_improved.py

Stage 3 controller. Identical to hybrid_robot.py but subclasses BraitenbergRobotImproved.

rl_env.py

SurveillanceEnv — step/reset interface. Reward: −dist penalty, −collision penalty, −time penalty, +capture bonus. Supports both robot classes via use_improved flag.

train_rl.py

REINFORCE with baseline. Analytic gradients (pure numpy chain rule). Curriculum: easy densities first. --improved flag for Stage 3.

Experiment runners and analysis

logger.py

Per-step CSV (positions, speeds, sensors, heading change, detection flag) + trial summary CSV. Fresh header written by runner, not by Logger.

run_experiments.py

Stage 1 headless batch runner. Generates fixed layouts via seed hierarchy. Also imported by Stages 2 and 3.

run_hybrid_experiments.py

Stage 2 evaluation. Loads trained policy, runs identical layouts as Stage 1.

run_improved_experiments.py

Stage 3 evaluation. Uses HybridRobotImproved with trained Stage 3 policy.

analyse.py

Reads one trials.csv, produces 6 plots: capture time boxplot, success rate, path ratio, pursuit efficiency, heading change rate, time to first detection.

compare.py

Reads all three trials.csv files, overlays all three controllers on 4 shared plots: success rate, capture time, collision count, pursuit efficiency.

06

Reinforcement learning — what and why

What the policy actually is

A two-layer neural network that takes the 12-dimensional sensor observation and outputs two numbers: Δα (adjustment to pursuit weight) and Δβ (adjustment to avoidance weight). These are added to the baseline values from config.py and clamped to [0.1, 2.5].

obs (12,) → Linear(12,32) → ReLU → actor: Linear(32,2) → tanh × 0.8 → (Δα, Δβ)
α_eff = clamp(1.0 + Δα, 0.1, 2.5)     β_eff = clamp(1.2 + Δβ, 0.1, 2.5)

The observation vector

[p0..p7, s_L, s_R, d_norm, θ_norm] — 8 proximity readings, left and right target sensor intensities, normalised distance to intruder, and normalised bearing to intruder. The policy sees exactly what the Braitenberg controller sees, nothing more.

Algorithm: REINFORCE with baseline

Each training episode: run the stochastic policy for up to 2000 steps, compute discounted returns G_t backwards through the episode, subtract the critic's value estimate V(s_t) to get advantages A_t = G_t − V(s_t), then update:

actor: W ← W + lr_actor · Σ_t A_t · ∇log π(a_t|s_t)
critic: W_v ← W_v − lr_critic · Σ_t (V(s_t) − G_t) · ∇V

All gradients are computed analytically by hand — chain rule through the tanh and linear layers in compute_gradients(). No external framework is used.

Reward function

r_t = −0.01 · d_norm − 0.5 · collided − 0.002 + 10.0 · captured

The dense distance penalty provides learning signal every step, not just on rare captures. The collision penalty and time penalty shape the tradeoff between aggressiveness and safety.

Curriculum

Episodes 1–500: only densities 0%, 5%, 10%. The policy first learns that pursuing the intruder is rewarding at all. Episodes 501+: full range 0%–40%. Without this, high-density environments produce near-zero reward gradient early in training.

What the policy learns At low density (clear sensors, target visible): output high Δα, moderate Δβ — pursue aggressively. At high density (proximity sensors reading high): output lower Δα, higher Δβ — prioritise avoidance. The policy is learning a function from sensor state to optimal α:β ratio. This is what the fixed controller cannot do.
07

Metrics and experimental design

Experimental structure

8 density levels × 4 fixed layouts × 5 trials = 160 trials per controller, 480 trials total across all three stages. All three controllers are evaluated on identical layouts and trial seeds. Statistical comparison uses Welch's t-test (p < 0.05) and Cohen's d for effect size.

Density levels

0%, 5%, 10%, 15%, 20%, 25%, 30%, 40% floor-area coverage. 0.25 was added specifically to resolve the performance cliff identified between 20% and 30% in early testing.

Five primary metrics

MetricDefinitionWhat it tells you
capture_rateFraction of trials where robot captured intruder before timeoutPrimary success measure
capture_stepSteps to capture; MAX_STEPS (4000) if timeoutSpeed of pursuit when successful
path_ratioActual path length / straight-line distance to intruder at trial endNavigation efficiency — how direct the route was
collision_countSteps where robot body overlapped an obstacleSafety — how often avoidance fails
pursuit_efficiency η1 / (capture_step × (1 + collisions) + ε)Composite — penalises both slow capture and high collisions

Two diagnostic metrics

MetricDefinitionWhat it diagnoses
time_to_first_detectionStep when either target sensor first read non-zero; −1 if neverDoes density delay target acquisition, or does the robot detect but fail to close?
mean_heading_change_rateMean |Δangle| per step in radiansHigh values = avoidance-channel thrashing. Low values at high density = stalling/freezing.

How to run the full pipeline

Install

pip install pygame matplotlib numpy

Stage 1 — baseline experiments

# Full 160-trial baseline run
python run_experiments.py

# Or with custom settings
python run_experiments.py --layouts 4 --trials 5 --out results/

Stage 2 — RL on original robot

# Train (500 episodes ≈ 25–35 min on a weak laptop)
python train_rl.py --episodes 500 --out models/stage2/

# Evaluate on same layouts as Stage 1
python run_hybrid_experiments.py --policy models/stage2/policy_final --out results_hybrid/

Stage 3 — RL on improved robot

# Train with --improved flag (same algorithm, different base robot)
python train_rl.py --episodes 500 --out models/stage3/ --improved

# Evaluate
python run_improved_experiments.py --policy models/stage3/policy_final --out results_improved/

Compare all three and produce plots

python compare.py \
  --baseline results/trials.csv \
  --hybrid results_hybrid/trials.csv \
  --improved results_improved/trials.csv \
  --out plots_compare/

Individual analysis

# Per-controller plots (6 plots each)
python analyse.py --csv results/trials.csv --out plots/baseline/
python analyse.py --csv results_hybrid/trials.csv --out plots/hybrid/
python analyse.py --csv results_improved/trials.csv --out plots/improved/

Live interactive simulation

python main.py   # S=sensors, R=reset, Space=pause, Q=quit

Run compare.py to generate the four comparison plots. These plots — especially the success rate comparison — are the central result of the project.

Fill out the LaTeX report

If there is time — the RL comparison baseline

The original plan included a fully trained deep RL agent as a comparison ceiling. This would use the same observation space and arena but replace the Braitenberg controller entirely, showing where the performance ceiling lies. This is noted in the report's future work section and can be added if time permits.

Report justifications already prepared

Several justifications were developed during the project that should be referenced explicitly in the report: the physical grounding of all parameters against the e-puck and TurtleBot3 platforms; the 2D top-down abstraction as standard in mobile robotics literature; the circular obstacle justification (columns, pillars, planters); the α:β ratio derivation from speed constraints; and the curriculum rationale for RL training. These are all documented in the conversation and partially included in the existing LaTeX.

Key finding to watch for The most interesting result will be whether Stage 2 (RL on original robot) shows a meaningful improvement over Stage 1 at medium densities (15%–25%), and whether Stage 3 recovers performance at high densities (30%–40%) where Stage 1 collapses to near-zero capture rate. If Stage 3 does not outperform Stage 2, the engineered improvements are doing the work — which is itself a finding worth reporting.
