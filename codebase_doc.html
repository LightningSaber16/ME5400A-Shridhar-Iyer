System Architecture

The codebase is split into three clean layers. The simulation layer handles all physics and agent logic with no display dependency. The visualisation layer reads state from the simulation and renders it to a Pygame window. The experiment layer drives the simulation headlessly in batch, logs results to CSV, and produces plots.

Key design principle The simulation logic (robot.py, intruder.py, geometry.py) contains zero Pygame imports. This means the same robot and intruder code runs identically in the live Pygame window and in headless batch experiments — no code duplication, no conditional imports.

config.py

Every other module imports this as import config as C and references values as C.ARENA_W, C.PURSUIT_WEIGHT, etc. This means any parameter change takes effect everywhere simultaneously — there are no magic numbers buried in other files.

§ 3

geometry.py — Math Utilities

geometry.py

A pure-math module with no simulation state and no dependencies beyond the standard library. All four functions are used by both robot.py and intruder.py. Keeping geometry isolated here means the math can be tested and reasoned about entirely on its own.

angle_wrap(a)

Wraps any angle to the range [−π, π] using atan2(sin(a), cos(a)). Used every time a bearing or heading change is computed to prevent accumulated drift and ensure correct left/right comparisons.

distance(ax, ay, bx, by)

Euclidean distance between two points via math.hypot. Called in collision detection, capture checking, and sensor intensity calculations.

angle_to(fx, fy, tx, ty)

Returns the bearing in radians from point F to point T using atan2(ty−fy, tx−fx). Used by the robot's target sensor to compute the intruder's bearing relative to the robot's heading.

ray_circle_intersection(ox, oy, dx, dy, cx, cy, r, max_dist)

Solves the analytic ray–circle intersection. The ray originates at (ox, oy) and travels in direction (dx, dy). Returns the distance to the nearest intersection with circle (cx, cy, r), or max_dist if no hit within range.

This is the core of the proximity sensor model. The quadratic discriminant is computed as:

disc = b² − 4ac   where   a = d·d,   b = 2(f·d),   c = f·f − r²

If disc < 0, no intersection. Otherwise the two roots t₁, t₂ are computed; the smallest positive t less than max_dist is returned.

ray_wall_intersection(ox, oy, dx, dy, arena_w, arena_h, max_dist)

Tests the ray against all four arena walls (x=0, x=W, y=0, y=H) using parametric line–plane intersection. Returns the closest wall hit within max_dist. This runs before the circle test in each sensor ray cast, so wall readings always take priority over obstacle readings at equal distance.

§ 4

robot.py — Braitenberg Vehicle

robot.py

The core of the project. BraitenbergRobot is a differential-drive agent whose motor commands are produced by superimposing two Braitenberg channels — no planning, no map, no memory beyond the current sensor readings.

Proximity sensing — _cast_proximity_sensors()

Eight rays are cast from the robot centre at angles 2πi/8 relative to the current heading for i = 0..7. For each ray, ray_wall_intersection is called first to find the nearest wall hit, then ray_circle_intersection is called for every obstacle, taking the minimum distance overall. The raw distance is converted to a normalised reading:

p_i = 1 − d_i / PROX_RANGE

A reading of 0 means the ray reached its full range without hitting anything. A reading of 1 means the obstacle is at contact distance. Readings are stored in robot.prox_readings and exposed for the renderer.

Target sensing — _sense_target()

The target sensor checks whether the intruder is within TARGET_SENSOR_RANGE and within ±TARGET_SENSOR_FOV of the robot's heading. If so, it computes an intensity that falls off with both distance and angular offset via cosine weighting:

s_L = (1 − d_t / d_tgt) · cos(|θ| / φ · π/2)   for θ ≥ 0 (target left of heading)

s_L and s_R are mutually exclusive — only the sensor on the same side as the target activates. When the target is exactly ahead, both sensors read equally. This smooth weighting prevents sudden jumps as the intruder crosses the heading axis.

Braitenberg control law — _braitenberg_control()

Two channels are superimposed. The pursuit channel implements Vehicle 2b (excitatory, crossed): each motor is driven by the opposite target sensor, causing the robot to steer toward the stimulus:

v_Lpursuit = α · g_t · s_R
v_Rpursuit = α · g_t · s_L

The avoidance channel implements Vehicle 3a (inhibitory): each body half aggregates its proximity load as the maximum reading across sensors in that hemisphere. A strong reading on the left inhibits the right motor, turning the robot away:

P_L = max(p_i : i ∈ left half)   P_R = max(p_i : i ∈ right half)
v_L = v_Lpursuit − β · g_p · P_R
v_R = v_Rpursuit − β · g_p · P_L

Final motor speeds are clamped to [0, MAX_SPEED]. Negative values are clipped to zero — the robot does not reverse in the baseline controller. No forward bias or motor floor is applied; the controller is a pure superposition of the two channels.

Differential-drive kinematics — _differential_drive()

Given left and right motor speeds (v_L, v_R), the forward velocity and angular velocity of the robot body are:

v_centre = (v_L + v_R) / 2     ω = (v_R − v_L) / wheelbase

The wheelbase is set to 2 × ROBOT_RADIUS (wheels at body edge). Heading and position are updated each step:

θ ← angle_wrap(θ + ω)     x ← x + v_centre · cos(θ)     y ← y + v_centre · sin(θ)

Collision resolution — _check_collision()

After every kinematic update, the robot's distance to every obstacle centre is checked. If the robot body overlaps an obstacle (dist < robot_radius + obstacle_radius), a push-out is applied along the collision normal to separate them. The robot.collided flag is set for that step and picked up by the logger and renderer.

intruder.py — Target Agent

intruder.py

The intruder is a circular agent that moves independently of the robot. It supports three motion modes, switchable at runtime via intruder.set_mode().

bounce mode (default)

The intruder moves at constant speed INTRUDER_SPEED in a fixed direction. On contact with a wall, the appropriate velocity component is negated (angle of incidence = angle of reflection). On contact with an obstacle, the reflection is computed from the collision normal using the formula:

θ_new = 2 · θ_normal − θ_current + π

random_walk mode

Same wall/obstacle reflection as bounce, but every _rw_interval steps (sampled uniformly from [30, 120]) the intruder adds a random heading perturbation of up to ±90°. This makes the target less predictable without making it fully erratic.

waypoints mode

The intruder steers directly toward a sequence of (x, y) waypoints, advancing to the next when within 20 px of the current target. The waypoint list is set via intruder.set_waypoints(points) and cycles indefinitely.

Note on arena clamping All three modes call _clamp_to_arena() at the end of each update to hard-clip the intruder position within the arena bounds. This prevents numeric drift from pushing the agent outside the walls in edge cases where the reflection logic doesn't fire exactly at the boundary.
§ 6

renderer.py — Pygame Visualisation

renderer.py

All drawing is handled by the Renderer class. It receives the current simulation state as arguments to draw() each frame — it holds no simulation state itself. This clean separation means the renderer can be dropped without touching any of the physics code.

An RGBA overlay Surface is used for transparent elements (sensor rays, sensor FOV wedge, capture ring). At each frame it is cleared, drawn to, and then composited onto the main screen via screen.blit(overlay). This lets sensor visualisations have variable opacity without affecting the solid obstacle and agent drawing beneath them.

Proximity sensor rays

Each of the 8 proximity rays is drawn as a line from the robot centre outward. The ray length is PROX_RANGE × (1 − reading) — a high reading (obstacle close) shortens the ray visually. Colour intensity increases with reading: a faint blue ray means clear space ahead, a bright one means the sensor is nearly at contact.

Target sensor FOV

The target sensor's field of view is drawn as two transparent amber wedges fanning left and right from the robot's heading direction. The arc edges are approximated with 20 line segments from the robot centre to points on the arc radius.

HUD overlay

A monospace readout in the top-left corner displays step count, robot position, motor speeds (v_L / v_R), and target sensor readings. A "TARGET CAPTURED" message is rendered centred on screen when captured=True.

main.py — Live Simulation Loop

main.py

The entry point for interactive use. Instantiates the renderer, robot, intruder, and logger, then runs the main Pygame event loop at FPS = 60 frames per second.

Event handling

KeyAction
SToggle sensor ray / FOV visualisation on/off
RReset robot, intruder, and logger to initial state
SpacePause / resume the simulation
Q / EscQuit

Obstacle loading

The load_obstacles() function in main.py returns the static layout from config.OBSTACLES by default. A commented block in the function shows how to replace this with a random density generator for manual interactive testing at specific densities.

Logger integration

A Logger instance is created at startup with density=0.0, trial=0. It logs every step to results/steps/ and writes a summary row to results/trials.csv on capture. On reset (R key), a new Logger instance is created fresh.

logger.py

§ 9

run_experiments.py — Headless Batch Runner

run_experiments.py

Drives the simulation headlessly — no Pygame window, no rendering — at maximum speed. This is the file you run to collect baseline data. The default configuration runs 160 trials (8 densities × 4 layouts × 5 trials).

Obstacle layout generation

Obstacles are placed by rejection sampling. A candidate circle is drawn at a random position and radius (15–30 px). It is rejected if it: (a) overlaps any existing obstacle with a small buffer gap, (b) falls within 60 px of the robot start position, or (c) falls within 50 px of the intruder start position. Sampling continues until the combined obstacle area reaches the target density fraction of the arena area.

Seed hierarchy — reproducibility

Every layout and trial is individually reproducible. The seed hierarchy is:

BASE_SEED = 42
layout_seed = BASE_SEED + density_index × 1000 + layout_id
→ determines obstacle positions for this layout
trial_seed = layout_seed × 100 + trial_index
→ determines intruder initial heading for this trial

Both the pure Braitenberg and hybrid experiments use the same seed hierarchy, so they are evaluated on identical layouts and intruder trajectories — a fair comparison.

analyse.py — Analysis and Plotting

analyse.py

Reads results/trials.csv and produces six plots saved to plots/. Requires matplotlib and numpy; falls back to a text summary if these are not installed.

Loading and cleaning

The CSV is read with encoding="utf-8-sig" to handle the BOM that Windows adds to text files. All string values are stripped of whitespace before numeric parsing — this prevents silent corruption from the \r line endings that Windows produces. Density keys are rounded to 4 decimal places before grouping to prevent floating-point split (0.1 vs 0.10000000002 landing in separate buckets).

End-to-End Data Flow

↳ creates fresh results/trials.csv (with header)
↳ for each density × layout × trial:
BraitenbergRobot.update() × MAX_STEPS
Intruder.update() × MAX_STEPS
Logger.step() × MAX_STEPS → results/steps/*.csv
Logger.close() → results/trials.csv (appended)

analyse.py
↳ reads results/trials.csv
↳ groups rows by density
↳ prints text summary table
↳ writes 6 plots → plots/*.png
Important run_experiments.py always creates a fresh trials.csv at the start of each run. If you want to preserve previous results, copy or rename the results/ directory before running again. The per-step CSVs in results/steps/ are never overwritten — filenames include density, layout id, and trial index.
